# 20260428.011 - Предотвращение повторных вызовов инструментов

**Дата:** 2026-04-28  
**Автор:** AI Agent (Claude)  
**Проблема:** Агент мог повторно вызывать одни и те же инструменты с одинаковыми параметрами

## Описание проблемы

При работе в несколько итераций агент мог повторно вызвать те же инструменты с теми же параметрами:

```
Итерация 1:
  action → exact_search(substring="ПО установлено")
  observation → "Найдено упоминание, но нет деталей"
  refine → needs_refinement=True

Итерация 2:
  action → exact_search(substring="ПО установлено")  ← ПОВТОР!
```

❌ **Проблемы:**
1. **Неэффективность** - повторный запрос к базе данных с теми же параметрами
2. **Потеря времени** - результаты уже есть в истории, но агент их игнорирует
3. **Риск зацикливания** - если агент не понимает, что нужно искать по-другому
4. **Отсутствие прогресса** - нет новой информации на следующей итерации

## Решение

Добавлена **явная передача истории выполненных инструментов** в промпт action node с инструкцией не повторять вызовы.

## Внесенные изменения

### 1. Промпт action/user.md

**Добавлен блок с историей выполненных инструментов:**

```jinja2
{% if all_tool_results %}
⚠️ **IMPORTANTE: Ранее выполненные инструменты (НЕ ПОВТОРЯЙ!):**

В предыдущих итерациях уже были выполнены следующие инструменты:
{% for tr in all_tool_results %}
{{ loop.index }}. {{ tr.tool }}({{ tr.input | tojson }})
{% endfor %}

**КРИТИЧНО:** НЕ повторяй вызовы с ТЕМИ ЖЕ параметрами!
- Результаты этих вызовов уже есть в истории сообщений
- Ищи ДРУГУЮ информацию, используй ДРУГИЕ параметры
{% endif %}
```

**Пример вывода:**
```
⚠️ **IMPORTANTE: Ранее выполненные инструменты (НЕ ПОВТОРЯЙ!):**

В предыдущих итерациях уже были выполнены следующие инструменты:
1. exact_search({"substring": "ПО установлено", "limit": 30})
2. semantic_search({"query": "АРМ оператора программное обеспечение", "top_k": 10})

**КРИТИЧНО:** НЕ повторяй вызовы с ТЕМИ ЖЕ параметрами!
```

### 2. Промпт action/system.md

**Добавлено критическое правило:**

```markdown
🚫 **КРИТИЧЕСКОЕ ПРАВИЛО: НЕ ПОВТОРЯЙ ВЫЗОВЫ!**
**ОБЯЗАТЕЛЬНО проверь список уже выполненных инструментов:**
- Если инструмент с ТАКИМИ ЖЕ параметрами уже был вызван - НЕ вызывай его снова!
- Результаты уже есть в истории сообщений
- Ищи информацию ДРУГИМИ способами:
  * Используй другой инструмент (semantic_search вместо exact_search)
  * Используй другие параметры (другие термины, другой раздел)
  * Если нашли упоминание раздела - читай его полностью

**Примеры правильного поведения:**
- ❌ НЕПРАВИЛЬНО: exact_search("ПО") → уже было → снова exact_search("ПО")
- ✅ ПРАВИЛЬНО: exact_search("ПО") → уже было → get_section_content(file="...", section="...")
- ✅ ПРАВИЛЬНО: semantic_search("АРМ") → уже было → exact_search_in_file_section(...)
```

### 3. Базовый системный промпт (system_prompt_base.md)

**Добавлен общий принцип:**

```markdown
🚫 **КРИТИЧЕСКОЕ ПРАВИЛО ИТЕРАЦИЙ:**
**НЕ повторяй вызовы инструментов с одинаковыми параметрами!**
- Результаты предыдущих вызовов уже есть в истории сообщений
- На каждой новой итерации ищи НОВУЮ информацию
- Используй ДРУГИЕ инструменты или ДРУГИЕ параметры
```

## Как это работает

### Workflow с предотвращением повторов:

```
Итерация 1:
  action → exact_search(substring="ПО установлено", limit=30)
  observation → "Найдено упоминание в разделе X"
  refine → needs_refinement=True

Итерация 2:
  user message → 
    "Ранее выполнены: exact_search({'substring': 'ПО установлено', 'limit': 30})"
    
  agent видит историю → НЕ вызывает exact_search снова
  
  action → get_section_content(source_file="...", section="X")  ✅ НОВЫЙ инструмент!
  observation → "Получен полный список ПО"
  refine → needs_refinement=False
```

## Преимущества

### 1. Эффективность

**До:**
- Повторные запросы к базе данных
- Лишние вызовы LLM
- Потеря времени на обработку

**После:**
- Каждый вызов инструмента приносит НОВУЮ информацию
- Нет дублирования работы
- Быстрее достижение результата

### 2. Прогресс между итерациями

**До:**
```
Итерация 1: exact_search → "нашли упоминание"
Итерация 2: exact_search → "нашли то же упоминание"  ← НЕТ ПРОГРЕССА!
```

**После:**
```
Итерация 1: exact_search → "нашли упоминание в разделе X"
Итерация 2: get_section_content(section=X) → "получили полный текст раздела"  ← ПРОГРЕСС!
```

### 3. Избежание зацикливания

Агент **вынужден** искать новую информацию другими способами, что предотвращает зацикливание на одних и тех же действиях.

### 4. Более глубокий анализ

Агент переходит от поверхностного поиска (exact_search) к глубокому анализу (get_section_content, exact_search_in_file_section).

## История вызовов доступна в state

Информация о всех предыдущих вызовах доступна через `state['all_tool_results']`:

```python
# В rag_lg_agent.py
state["all_tool_results"] = [
    {
        "tool": "exact_search",
        "input": {"substring": "ПО", "limit": 30},
        "result": "..."
    },
    {
        "tool": "semantic_search",
        "input": {"query": "АРМ", "top_k": 10},
        "result": "..."
    },
    # ...
]
```

Эта информация передается в Jinja2 шаблон и отображается в user message для action node.

## Формат вывода в промпте

```jinja2
{% for tr in all_tool_results %}
{{ loop.index }}. {{ tr.tool }}({{ tr.input | tojson }})
{% endfor %}
```

Выводит:
```
1. exact_search({"substring": "ПО", "limit": 30})
2. semantic_search({"query": "АРМ оператора", "top_k": 10})
3. get_section_content({"source_file": "file.md", "section": "Section"})
```

Агент видит **точные параметры** каждого вызова и может сравнить их с тем, что планирует вызвать.
