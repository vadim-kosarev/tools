# 20260428.014 - Упрощение форматирования вопросов в user messages

**Дата:** 2026-04-28  
**Автор:** AI Agent (Claude)  
**Проблема:** LLM игнорировал вопрос пользователя даже когда он был в промпте

## Описание проблемы

После удаления конкретных примеров, LLM (qwen3.5:9b) все равно не видел вопрос:

```bash
python rag_lg_agent.py "что такое smart monitor"

# Response:
{"thought": "Пользователь не сформулировал вопрос. Необходимо запросить уточнение."}
```

### Проверка логов

В логе `002_llm_plan_request.log` вопрос **ПРИСУТСТВОВАЛ**:

```
[#1 USER]
🎯 **РЕАЛЬНЫЙ ВОПРОС ПОЛЬЗОВАТЕЛЯ:**

что такое smart monitor

⚠️ Отвечай на ЭТОТ вопрос, а не на примеры из system message!
```

### Корневая причина

Сложное форматирование с **эмодзи** (🎯, ⚠️) и **жирным шрифтом** (**РЕАЛЬНЫЙ ВОПРОС**) могло восприниматься LLM как:
- Служебная информация
- Разметка промпта
- Не реальный вопрос

LLM (особенно более слабые модели как qwen3.5:9b) могут **неправильно интерпретировать** такое форматирование.

## Решение

**УПРОЩЕНО форматирование** всех user.md до минимума:

```markdown
Вопрос пользователя:
{{ user_query }}
```

### 3. observation/user.md, refine/user.md, final/user.md

Аналогичные изменения - убраны эмодзи, жирный шрифт и предупреждения.

### 4. plan/system.md

**Удалены последние конкретные примеры:**

**Стало:**
```markdown

- Комбинации терминов с их синонимами
```

## Почему это работает

### Принцип "KISS" (Keep It Simple, Stupid)

**Сложное форматирование:**
```
🎯 **РЕАЛЬНЫЙ ВОПРОС ПОЛЬЗОВАТЕЛЯ:**
что такое smart monitor
⚠️ Отвечай на ЭТОТ вопрос!
```

LLM думает:
- "Это какая-то разметка?"
- "Эмодзи = служебная информация?"
- "РЕАЛЬНЫЙ ВОПРОС = не реальный, а пример?"

**Простое форматирование:**
```
Вопрос пользователя:
что такое smart monitor
```

LLM думает:
- "Понятно - это вопрос"
- "Нужно на него ответить"

### Совместимость с разными LLM

**Сложные модели** (GPT-4, Claude) могут понимать эмодзи и сложное форматирование.

**Простые модели** (qwen, llama, mistral) лучше работают с **простым текстом**.

Упрощение делает промпты **универсальными**.

## Дополнительные преимущества

### 1. Меньше токенов

**До:**
```
🎯 **РЕАЛЬНЫЙ ВОПРОС ПОЛЬЗОВАТЕЛЯ:**
{{ user_query }}
⚠️ Отвечай на ЭТОТ вопрос, а не на примеры из system message!
```
~30 токенов служебной информации

**После:**
```
Вопрос пользователя:
{{ user_query }}
```
~5 токенов

**Экономия:** ~25 токенов на каждый вызов × 5 nodes = ~125 токенов на запрос

### 2. Быстрее генерация

Меньше токенов → быстрее обработка промпта → быстрее ответ

### 3. Меньше путаницы

Нет лишних предупреждений и маркеров → LLM концентрируется на сути

### 4. Проще отладка

Логи читаются проще без куч эмодзи и предупреждений

## Тестирование

### Проверка 1: Базовый вопрос

```bash
python rag_lg_agent.py "что такое smart monitor"
```

**Ожидается:**
```json
{
  "thought": "Пользователь спрашивает что такое 'smart monitor'",
  "plan": [...]  // План поиска по этому термину
}
```

**НЕ должно быть:**
```json
{
  "thought": "Пользователь не сформулировал вопрос"  // ❌
}
```

### Проверка 2: Сложный вопрос

```bash
python rag_lg_agent.py "Какое ПО установлено на АРМ оператора"
```

**Ожидается:**
```json
{
  "thought": "Пользователь спрашивает какое ПО установлено на АРМ оператора",
  "plan": [...]  // План с multi_term_exact_search
}
```

### Проверка 3: Короткий вопрос

```bash
python rag_lg_agent.py "серверы"
```

**Ожидается:**
```json
{
  "thought": "Пользователь спрашивает про серверы",
  "plan": [...]  // План поиска информации о серверах
}
```
