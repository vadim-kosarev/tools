# Отчет: Устранение дублирования логирования инструментов

**Дата:** 2026-04-28  
**Тип изменений:** Рефакторинг системы логирования  
**Затронутые файлы:** `RAG/llm_call_logger.py`, `RAG/kb_tools.py`

---

## Проблема

Логирование вызовов инструментов происходило дважды:

1. **Внутри каждого инструмента** (`kb_tools.py`):
   - Создание args_dict
   - Вызов `_db_request()`
   - Вызов `rec.set_response()` для логирования результата

2. **На уровне LangChain** (`llm_call_logger.py`):
   - `on_tool_start()` - автоматически логирует вызов с аргументами
   - `on_tool_end()` - автоматически логирует результат

Это приводило к:
- ❌ Дублированию логов
- ❌ Загромождению кода
- ❌ Избыточности поддержки
- ❌ Путанице при анализе логов

---

## Решение

### 1. Улучшен LangChain callback handler

**Файл:** `llm_call_logger.py`

**Изменения в `on_tool_start()`:**

```python
def on_tool_start(self, serialized: dict[str, Any], input_str: str, *, run_id: UUID, **kwargs: Any) -> None:
    if not self._log._enabled:
        return
    tool_name = serialized.get("name", "tool")
    
    # ✅ НОВОЕ: Попытка отформатировать input_str как JSON для читаемости
    formatted_input = input_str
    try:
        # Попробуем распарсить как JSON и красиво отформатировать
        parsed = json.loads(input_str)
        formatted_input = json.dumps(parsed, ensure_ascii=False, indent=2)
    except (json.JSONDecodeError, TypeError):
        # Если не JSON - оставляем как есть
        pass
    
    rec = self._log.start_record(f"TOOL:{tool_name}")
    rec.set_request(formatted_input)
    self._tool_records[str(run_id)] = (rec, formatted_input)
```

**Что это дает:**
- ✅ Автоматическое форматирование JSON аргументов с отступами
- ✅ Корректное отображение кириллицы (`ensure_ascii=False`)
- ✅ Fallback на оригинальный формат если не JSON

### 2. Удалено дублирующее логирование из kb_tools.py

**Удалено:**

1. **Функция `_format_tool_args()`** - больше не нужна
2. **Функция `_db_request()`** - больше не нужна
3. **~180 строк кода** создания `args_dict` в каждом инструменте
4. **Все вызовы `rec = _db_request(...)`** (16 инструментов)
5. **Все блоки `if rec: rec.set_response(...)`** (16 инструментов)

**Пример было/стало:**

#### Было (дублирование):

```python
@tool
def exact_search(substring: str, limit: int = 30, ...) -> SearchChunksResult:
    logger.debug(f"Tool exact_search: substring='{substring}'...")
    
    # 📝 Формируем JSON для логирования
    args_dict = {
        "substring": substring,
        "limit": limit,
        "chunk_type": chunk_type
    }
    if source:
        args_dict["source"] = source
    if section:
        args_dict["section"] = section
    
    # 📝 Логируем аргументы
    rec = _db_request("DB:exact_search", _format_tool_args(**args_dict))
    
    # Основная логика
    docs = vectorstore.clone().exact_search(substring, limit=limit, ...)
    chunks = _deduplicate_chunks(_docs_to_chunk_results(docs))
    result = SearchChunksResult(query=substring, chunks=chunks, total_found=len(chunks))
    
    # 📝 Логируем результат
    if rec:
        rec.set_response(f"Найдено {len(chunks)} чанков\n\n{result.model_dump_json(indent=2)}")
    logger.info(f"exact_search '{substring}': {len(chunks)} чанков")
    
    return result
```

#### Стало (без дублирования):

```python
@tool
def exact_search(substring: str, limit: int = 30, ...) -> SearchChunksResult:
    logger.debug(f"Tool exact_search: substring='{substring}'...")
    
    # Основная логика
    docs = vectorstore.clone().exact_search(substring, limit=limit, ...)
    chunks = _deduplicate_chunks(_docs_to_chunk_results(docs))
    result = SearchChunksResult(query=substring, chunks=chunks, total_found=len(chunks))
    
    logger.info(f"exact_search '{substring}': {len(chunks)} чанков")
    
    return result
```

**Что удалено:** ~14 строк избыточного кода на инструмент × 16 инструментов = **~224 строки**

## Как теперь работает логирование

### 1. Автоматическое логирование через LangChain

Когда RAG агент вызывает инструмент:

```python
# RAG агент
result = exact_search.invoke({"substring": "MongoDB", "limit": 30})
```

LangChain автоматически:

1. **Вызывает `on_tool_start()`**:
   - Логирует `TOOL:exact_search` REQUEST
   - Форматирует аргументы в JSON с отступами
   - Записывает в файл `NNN_tool_exact_search_request.log`

2. **Выполняет инструмент**

3. **Вызывает `on_tool_end()`**:
   - Логирует `TOOL:exact_search` RESPONSE
   - Включает args + результат
   - Записывает в файл `NNN_tool_exact_search_response.log`

### 2. Формат логов

**REQUEST (автоматически форматируется):**
```
#015  2026-04-28 10:00:00  [TOOL:exact_search]  REQUEST
{
  "substring": "MongoDB",
  "limit": 30,
  "chunk_type": ""
}
#015  2026-04-28 10:00:01  [TOOL:exact_search]  REQUEST end
```

**RESPONSE (автоматически включает args):**
```
#016  2026-04-28 10:00:05  [TOOL:exact_search]  RESPONSE
[args]
{
  "substring": "MongoDB",
  "limit": 30
}
----------------------------------------
{"query":"MongoDB","chunks":[...],"total_found":15}
#016  2026-04-28 10:00:06  [TOOL:exact_search]  RESPONSE end
```

## Преимущества изменений

### ✅ Устранено дублирование
- Логирование происходит в одном месте (LangChain callback)
- Нет конфликта между DB: и TOOL: префиксами
- Чище логи без повторяющейся информации

### ✅ Упрощен код
- **Удалено 2 функции** (`_format_tool_args`, `_db_request`)
- **Удалено ~224 строки** избыточного кода
- **16 инструментов** стали значительно проще

### ✅ Лучшее форматирование
- JSON аргументы автоматически форматируются с отступами
- Кириллица отображается корректно
- Fallback на текстовый формат если не JSON

### ✅ Проще поддержка
- Добавление нового инструмента не требует логирования
- Изменение формата логов в одном месте (callback handler)
- Меньше мест для возможных ошибок

### ✅ Единообразие
- Все инструменты логируются одинаково
- Одинаковый формат для LLM и TOOL вызовов (только префикс отличается)
- Консистентная структура логов

### ⚠️ Breaking changes

1. **Префикс логов изменился:**

   - Стало: `TOOL:exact_search`

2. **Логи больше не дублируются:**
   - Был только один REQUEST/RESPONSE на вызов
   - Нет отдельных DB: логов

**Миграция парсеров:**
