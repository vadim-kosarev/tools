## Breaking change: Postgres 18 volume mount

Официальный образ `postgres:18+` требует монтировать volume на `/var/lib/postgresql` целиком
(совместимость с `pg_ctlcluster`), а не на `/var/lib/postgresql/data`, как было. Со старым маунтом
контейнер падал в рестарт-луп с ошибкой "these Docker images are configured to store database
data in a format which is compatible with pg_ctlcluster...".

**Фикс:** в `docker-compose.yml` сервис `postgres` →
```yaml
volumes:
  - frigate_postgres_data:/var/lib/postgresql   # было: /var/lib/postgresql/data
  - ./postgres-init:/docker-entrypoint-initdb.d
```
Так как volume был свежий и пустой (первый запуск на starlight), пересоздали без потери данных.

## Найден и исправлен баг в `postgres-init/02.tables.sql` (не связан с версией Postgres)

Таблица `data_anpr` (со внешним ключом на `data_events`) создавалась в файле **раньше**, чем сама
`data_events` — FK падал с `ERROR: relation "data_events" does not exist`, `ON_ERROR_STOP`
обрывал выполнение всего файла, и в базе не оставалось ни одной таблицы.

Причина — это дамп из HeidiSQL (MySQL-ориентированный клиент), таблицы в файле шли по алфавиту
(`data_anpr` < `data_events`), из-за чего и порядок оказался сломан. Баг наверняка есть и в
исходном дампе на brightsky тоже, просто там никогда не стартовали с полностью чистого volume,
чтобы его словить.

**Исправлено:**
1. Переставлен блок `CREATE TABLE "data_events"` перед `CREATE TABLE "data_anpr"`.
2. После этого вскрылись ещё два MySQL-изма от того же HeidiSQL-экспорта, которые тоже не дают
   скрипту доехать до конца на Postgres:
   - `INDEX "idx_events_source" (...)` / `UNIQUE INDEX "uq_data_events_data_hash" (...)` **внутри**
     `CREATE TABLE data_events` — в Postgres такого синтаксиса нет, индексы создаются отдельными
     `CREATE INDEX` / `CREATE UNIQUE INDEX` после таблицы. Вынесены в отдельные statements.
   - `) ENGINE=MyISAM;` на временных таблицах `view_events_media` / `view_plate_numbers` —
     тоже чисто MySQL-синтаксис, невалиден в Postgres. Убран.

После правок `02.tables.sql` отрабатывает целиком без ошибок: 4 таблицы, 2 индекса, 2 view.
Проверено пересозданием volume с нуля.

**Исправлено и это:** таблицы реально создавались в базе `rgzz` (дефолтная БД по
`POSTGRES_USER=rgzz`, т.к. `POSTGRES_DB` не был задан), а не в `frigate`, которую создавал
`01-init.sql` — `\c frigate` там действовал только в рамках его собственной сессии, `02.tables.sql`
подключался заново к дефолтной БД.

Добавлено `POSTGRES_DB: frigate` в `environment` сервиса `postgres` — теперь официальный образ сам
создаёт базу `frigate` при первом старте и подключает к ней все init-скрипты. Из-за этого пришлось
убрать ручной `CREATE DATABASE frigate;` из `01-init.sql` (иначе конфликт "database already
exists" и обрыв скрипта) — оставлен только `ALTER DATABASE frigate SET timezone = ...`.

Проверено пересозданием volume с нуля: `01-init.sql` и `02.tables.sql` отрабатывают полностью
без ошибок, `\dt`/`\dv` в базе `frigate` показывают все 2 таблицы + 2 view; дефолтной базы `rgzz`
больше не существует вовсе (`FATAL: database "rgzz" does not exist` — ожидаемо и верно).

**Ещё не тронуто:** `dump.sql` (третий файл в `postgres-init/`) — использует MySQL-синтаксис
(`DELIMITER //`) для переопределения функций pgcrypto (`armor`, `crypt`, `decrypt` и т.д.),
невалиден в Postgres целиком, падает на первом же выражении. Похоже на лишний/сломанный файл —
эти функции и так предоставляются расширением `pgcrypto` (`CREATE EXTENSION pgcrypto`). Не входило
в задачу (просили только порядок таблиц в `02.tables.sql`), но стоит иметь в виду.

## Проблема 1: Frigate грузит CPU на 100%, GPU почти не используется

Обнаружено пользователем после запуска: детекция объектов идёт на **CPU-детекторе** (TFLite SSD
MobileNet), а не на GPU — как и задокументировано в `README.md` ("ONNX-детектор с TensorRT не
использовать — компиляция engine при старте зависает"). GPU при этом занят только под
hwaccel-декодирование видео (`dec: ~13%`), не под инференс.

### Попытка включить GPU (ONNX-детектор)

Модель (`config/onnx/yolov9t_transposed.onnx`) и лейблмап уже были подготовлены заранее — в
`config.yaml` детектор/модель были закомментированы, а путь к лейблмапу указывал на
несуществующий `/config/yyy/coco-80.txt`. Внутри самого образа `frigate:stable-tensorrt` реально
лежит `/labelmap/coco-80.txt` — поправил путь на него и раскомментировал:
```yaml
detectors:
  onnx:
    type: onnx
model:
  model_type: yolo-generic
  input_tensor: nchw
  input_dtype: float
  labelmap_path: /labelmap/coco-80.txt
  path: /config/onnx/yolov9t_transposed.onnx
  width: 640
  height: 640
```

**Результат — детектор упал в crash-loop** (перезапуск каждые ~34 сек, `detection_fps: 0`,
детекция полностью не работает):
```
[E:onnxruntime:, inference_session.cc:2094 Initialize] This session cannot use the graph capture
feature as requested by the user as all compute graph nodes have not been partitioned to the
CUDAExecutionProvider
```
Перед этим ещё warning: `14 Memcpy nodes are added to the graph ... for CUDAExecutionProvider`
и `Error merging shape info for output. 'output0' source:{1,8400,84} target:{1,84,8400}`.

**Диагноз:** модель `yolov9t_transposed.onnx` (получена через `models/transpon_model.py` —
ручное добавление Transpose-узла в конец графа исходной `yolov9t.onnx`) не полностью
партиционируется на CUDAExecutionProvider — часть операций (видимо, из-за добавленного вручную
узла) уходит на CPU, из-за чего появляются Memcpy-узлы между провайдерами. Оптимизированный
runner Frigate (`get_optimized_runner`, ONNX-детектор в 0.17.2) пытается включить CUDA graph
capture, который требует, чтобы **весь** граф выполнялся на одном EP — и падает вместо
грейсфул-фолбэка. Это подтверждает предупреждение из `README.md`, только форма отказа другая
(раньше — зависание на компиляции engine, сейчас — мгновенный crash-loop), суть та же:
эта конкретная модель несовместима с GPU-путём в текущей версии Frigate.

**Откачено обратно** на CPU-детектор (закомментировал `detectors`/`model` в `config.yaml` —
проверено, детекция снова работает, `frigate.detectors WARNING: CPU detectors are not
recommended...`, `TensorFlow Lite XNNPACK delegate for CPU` создан, камеры отдают кадры).

### Что нужно, чтобы реально включить GPU-детекцию (не сделано)

Проблема в самой модели, не в конфиге. Варианты:
- Экспортировать ONNX-модель йолы без ручной пост-обработки графа (без append Transpose-узла
  через `onnx.helper`) — то есть найти/собрать модель, которая нативно отдаёт форму
  `(1, 8400, 84)`, а не патчить готовую `(1, 84, 8400)`.
- Либо попробовать уже лежащую в репо `models/yolov9s.onnx` (другая модель, не патченная
  `transpon_model.py`) — не проверяли, не факт что она в нужной форме на выходе.
- Либо использовать одну из моделей, официально рекомендованных в доках Frigate для ONNX-детектора
  (`yolonas`, `yolox`, `dfine` и т.д.) — они тестируются апстримом именно под этот
  optimized-runner путь.

## Проблема 1 — реальная причина и фикс: `.wslconfig` резал ядра, не Frigate

После отката GPU-детектора пользователь заметил: на brightsky Frigate работает на CPU/GPU
33%/15%, а на starlight — 85%/3%. Причина оказалась не в конфиге Frigate, а в Docker Desktop /
WSL2 на хосте starlight:

- Хост: AMD Ryzen 7 5700G — 8 ядер / 16 потоков, общая загрузка хоста в моменте всего 13-26%.
- `C:\Users\Vadim\.wslconfig` содержал `processors=4` — WSL2 (а значит и все Docker-контейнеры)
  были жёстко ограничены 4 виртуальными ядрами, хотя 12 потоков простаивали.
- Проверено: `nproc` на хосте — 16, `docker run --rm alpine nproc` — 4.

**Фикс:** `processors=4` → `processors=12` в `.wslconfig` (4 потока намеренно оставлены под
Windows-десктоп, за которым тут же работают). Применено через `wsl --shutdown` + автоматический
рестарт Docker Desktop.

**Результат:**
- `docker run --rm alpine nproc` теперь показывает `12`.
- Frigate: CPU **85% → 19%**, GPU 3% → **18%** (детектор по-прежнему CPU/TFLite — просто теперь
  считает свою нагрузку относительно 12 ядер вместо 4).
- Все контейнеры на хосте (`immich_postgres`, `rgzz.process_anpr`, `rgzz.track_mqtt`, `clickhouse`,
  оба Portainer) поднялись сами за ~8 сек (`restart: unless-stopped`), без потерь.
- `immich_postgres` standby-реплика восстановила стриминг с brightsky сразу после рестарта
  (`status: streaming`, `sender_host: 192.168.1.43`) — без пересидинга.

Итоговое соотношение CPU/GPU на starlight (19%/18%) теперь даже лучше, чем на brightsky (33%/15%).
