# Унификация параметров инструментов (query/substring)

**Дата:** 2026-04-29  
**Тип работы:** Улучшение UX для LLM

## 📋 Задача

Упростить работу LLM с инструментами поиска, унифицировав названия параметров:
- `semantic_search` использовал параметр `query`
- `exact_search` использовал параметр `substring`

LLM путался в этих названиях, что приводило к ошибкам.

## ✅ Решение

### 1️⃣ Расширена прослойка `_fix_tool_args`

**Файл:** `rag_lg_agent.py`

Добавлена автоматическая конвертация параметров:

```python
def _fix_tool_args(tool_name: str, tool_input: dict) -> dict:
    """Исправляет типичные ошибки LLM при именовании параметров."""
    fixed = tool_input.copy()
    
    # semantic_search: принимает 'query', но если передан 'substring', конвертируем
    if tool_name == "semantic_search":
        if "substring" in fixed and "query" not in fixed:
            fixed["query"] = fixed.pop("substring")
    
    # exact_search: принимает 'substring', но если передан 'query', конвертируем
    if tool_name == "exact_search":
        if "query" in fixed and "substring" not in fixed:
            fixed["substring"] = fixed.pop("query")
    
    # ...existing code...
```

### 2️⃣ Упрощен промпт ACTION

**Файл:** `prompts_v2/action/user.md`

**Было:**
```markdown
- **Точность параметров**: 
  - `semantic_search` использует параметр **`query`** (текстовый запрос для семантического поиска)
  - `exact_search` использует параметр **`substring`** (точная подстрока для поиска)
  - ⚠️ **НЕ путай** query и substring - это разные параметры для разных инструментов!
```

**Стало:**
```markdown
- **Точность параметров**: 
  - `semantic_search` - семантический поиск по смыслу (параметр: `query` или `substring`)
  - `exact_search` - точный поиск по подстроке (параметр: `query` или `substring`)
  - 💡 Можешь использовать любое название - система автоматически конвертирует
```

## 💡 Архитектурное решение

Использована **существующая прослойка** `_fix_tool_args`, которая уже исправляла другие ошибки LLM:
- `term` → `substring` для `find_sections_by_term`
- `source`/`file` → `source_file` для поисковых инструментов

Теперь она также исправляет путаницу между `query` и `substring`.

Это **лучше**, чем менять сигнатуры самих инструментов, потому что:
1. Не ломает существующий код
2. Сохраняет семантическую ясность (в инструменте используется правильное название)
3. Прозрачно для LLM - он может использовать любое название
4. Централизована логика фиксинга в одном месте
