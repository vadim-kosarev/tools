# Дедупликация чанков в Analyzer по chunk_id

**Дата:** 2026-04-29  
**Тип работы:** Оптимизация работы агента

## 📋 Задача

Добавить дедупликацию чанков в узел `analyzer` для устранения дубликатов, которые могут появиться при выполнении нескольких инструментов поиска.

**Проблема:**
- При выполнении плана агент может вызвать несколько инструментов:
  - `semantic_search(query="СУБД")`
  - `exact_search(substring="PostgreSQL")`
  - `find_sections_by_term(substring="база данных")`
- Один и тот же чанк может попасть в результаты нескольких инструментов
- Это увеличивает контекст для LLM и снижает эффективность

## ✅ Решение

### 1️⃣ Создана функция `_deduplicate_chunks_by_id`

**Файл:** `rag_lg_agent.py`

```python
def _deduplicate_chunks_by_id(context: list[dict]) -> list[dict]:
    """
    Дедуплицирует чанки по chunk_id в результатах всех инструментов.
    
    Просматривает все результаты типа SearchChunksResult в context
    и удаляет дубликаты по metadata.chunk_id, оставляя первое вхождение
    (с лучшим score, если есть).
    """
```

**Логика:**
- Проходит по всем результатам инструментов в `context`
- Для результатов типа `SearchChunksResult` (с полем `chunks`):
  - Извлекает `chunk_id` из метаданных каждого чанка
  - Оставляет только первое вхождение каждого `chunk_id`
  - Обновляет `total_found` и `result_md`
- Для остальных результатов (ошибки, не-SearchChunksResult) оставляет как есть

**Приоритет:** Первое вхождение = лучший score (т.к. инструменты выполняются по порядку важности)

### 2️⃣ Интегрирована в `analyzer_node`

**Применение дедупликации:**
```python
# Собираем context из tool execution записей
context = [ ... ]

# Дедупликация чанков по chunk_id
chunks_before = sum(len(e.get("result", {}).get("chunks", [])) for e in context ...)
context = _deduplicate_chunks_by_id(context)
chunks_after = sum(len(e.get("result", {}).get("chunks", [])) for e in context ...)

dedup_stats = f"Дедупликация: {chunks_before} → {chunks_after} чанков ({chunks_before - chunks_after} дубликатов удалено)"
```

### 3️⃣ Добавлена статистика в логи

**В логах теперь видно:**
```

Элементов для анализа: 3
Дедупликация: 25 → 18 чанков (7 дубликатов удалено)

## 📊 ANALYZER COMPLETE
Выполнено инструментов: 3 (успешно: 3, ошибок: 0)

```

**В консоли:**
```

```

## 🎯 Результаты

### Преимущества:

✅ **Уменьшение контекста** - удаляются дубликаты чанков из разных инструментов  
✅ **Экономия токенов** - меньше повторяющейся информации для LLM  
✅ **Улучшение качества** - LLM видит уникальные факты без повторений  
✅ **Прозрачность** - статистика дедупликации в логах и консоли  
✅ **Сохранение порядка** - первое вхождение (с лучшим score) сохраняется  

### Технические детали:

- **Работает с:** `SearchChunksResult` (semantic_search, exact_search, etc.)
- **Не влияет на:** Другие типы результатов (списки разделов, get_section_content, etc.)
- **Ключ дедупликации:** `metadata.chunk_id` (UUID)
- **Обновление:** Автоматически пересчитывает `total_found` и `result_md`

## 📈 Влияние на архитектуру

Дедупликация происходит в узле **analyzer** после выполнения всех инструментов на текущей итерации:

```
tool_executor → analyzer [дедупликация здесь] → refiner → final
```

**Почему в analyzer, а не в tool_executor?**
1. Дубликаты возникают между разными инструментами, а не внутри одного
2. Analyzer - логическое место для анализа и очистки данных
3. Не влияет на работу отдельных инструментов
4. История (`history`) хранит исходные результаты, дедуплицируется только `context`

## 🔄 История и контекст

**Важно:** Дедупликация применяется только к `context` (для финального ответа).

- **`history`** - полная история с дубликатами (для LLM контекста и отладки)
- **`context`** - дедуплицированные результаты (для эффективного финального ответа)

Это позволяет:
- LLM видит полную картину работы агента (в истории)
- Финальный ответ формируется на основе уникальных данных (в контексте)

## 📝 Будущие улучшения

Потенциальные расширения функции `_deduplicate_chunks_by_id`:

1. **Умное слияние scores** - если чанк найден несколькими инструментами, можно:
   - Взять минимальный score (лучшая семантическая близость)
   - Создать composite score (комбинация semantic + exact match)
   
2. **Ранжирование по типу инструмента** - приоритет:
   - 1. exact_search (точное совпадение)
   - 2. semantic_search (семантическая близость)
   - 3. find_sections_by_term (широкий поиск)

3. **Статистика по инструментам** - показать, какие инструменты дали больше всего дубликатов

4. **Настройка через ENV** - флаг `ENABLE_CHUNK_DEDUPLICATION=true/false`
