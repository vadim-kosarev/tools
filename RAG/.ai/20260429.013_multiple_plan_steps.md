# Множественное выполнение шагов плана в tool_selector

**Дата:** 2026-04-29  
**Тип работы:** Улучшение архитектуры агента

## 📋 Задача

Улучшить tool_selector для выполнения нескольких шагов плана за одну итерацию, если они могут быть выполнены параллельно.

**Проблема (было):**
- tool_selector выбирал инструменты только для одного шага плана за итерацию
- Даже если несколько шагов можно было выполнить параллельно, они выполнялись последовательно
- Это приводило к лишним итерациям и замедлению работы агента

**Цель:**
- tool_selector должен уметь выбирать инструменты для нескольких последовательных шагов плана
- Передавать информацию о выполняемых шагах в flow
- refiner должен понимать, какие шаги были выполнены

## ✅ Решение

### 1️⃣ Обновлена модель AgentAction

**Файл:** `rag_lg_agent.py`

**Добавлено поле `plan_steps`:**
```python
class AgentAction(BaseModel):
    """Ответ tool_selector-а."""
    status: Literal["action"] = Field(default="action")
    step: int = Field(description="Номер шага")
    thought: str = Field(description="Краткое рассуждение")
    actions: list[ToolAction] = Field(description="Список вызовов инструментов")
    plan_steps: list[int] = Field(
        description="Индексы шагов плана (0-based), которые покрываются этими инструментами. Например: [0, 1] если выполняются шаги 1 и 2"
    )
```

### 2️⃣ Расширено состояние AgentState

**Добавлены поля:**
```python
class AgentState(TypedDict, total=False):
    # ...existing code...
    completed_steps: list[int]         # список индексов завершенных шагов плана (0-based)
    executing_steps: list[int]         # список индексов шагов, выполняемых в текущей итерации (0-based)
```

### 3️⃣ Модифицирован tool_selector_node

**Изменения:**
- Передает в промпт информацию о `completed_steps` и `available_steps`
- Сохраняет `plan_steps` из результата LLM в `executing_steps`
- Логирует выполняемые шаги с их описанием

**Пример лога:**
```
## 🔧 TOOL_SELECTOR COMPLETE
Thought: Выполняю шаги 1 и 2: поиск СУБД и IP-адресов одновременно
Executing plan steps: #1: Найти все СУБД, #2: Найти IP-адреса
Actions:
  - semantic_search({"query": "СУБД базы данных"})
  - regex_search({"pattern": "\\d+\\.\\d+\\.\\d+\\.\\d+"})
```

### 4️⃣ Модифицирован refiner_node

**Изменения:**
- Получает `executing_steps` из состояния
- Проверяет выполнение всех executing_steps
- При успехе добавляет executing_steps в completed_steps
- Находит следующий невыполненный шаг как `current_step_index`

**Логика:**
```python
# Если текущие шаги выполнены, добавляем их в completed_steps
if result.current_step_completed:
    for step_idx in executing_steps:
        if step_idx not in completed_steps:
            completed_steps.append(step_idx)
    completed_steps.sort()

# Находим следующий невыполненный шаг
next_step_index = next((i for i in range(len(plan)) if i not in completed_steps), len(plan))
```

**Пример лога:**
```
## 🎯 REFINER START
Выполняемые шаги: #1: Найти все СУБД, #2: Найти IP-адреса
Завершенные шаги: #1, #2
Собрано результатов: 25

## 🎯 REFINER COMPLETE
Решение: → tool_selector
Причина: Переход к шагу #3: Получить описание (1 шагов осталось)
Завершено шагов: 2/3
```

## 🎯 Результаты

### Преимущества:

✅ **Меньше итераций** - несколько шагов выполняются за раз  
✅ **Быстрее работа** - параллельное выполнение вместо последовательного  
✅ **Умный выбор** - LLM сам решает, какие шаги можно объединить  
✅ **Прозрачность** - видно какие шаги выполняются и завершены  
✅ **Гибкость** - можно выполнять 1, 2, 3+ шагов за итерацию  

### Сценарии использования:

**Сценарий 1: Параллельный поиск**
```
План:
1. Найти все СУБД
2. Найти IP-адреса
3. Получить описание

Итерация 1: Шаги 1 и 2 → semantic_search + regex_search
Итерация 2: Шаг 3 → get_section_content
```

**Сценарий 2: Последовательные зависимые шаги**
```
План:
1. Найти разделы по теме
2. Прочитать содержимое найденных разделов
3. Извлечь ключевые факты

Итерация 1: Шаг 1 → find_sections_by_term
Итерация 2: Шаг 2 → get_section_content (зависит от результата шага 1)
Итерация 3: Шаг 3 → анализ
```

## 🔬 Технические детали

### Отслеживание состояния:

**completed_steps** - список завершенных шагов:
```python
[0, 1]  # Шаги #1 и #2 завершены
```

**executing_steps** - шаги в текущей итерации:
```python
[2, 3]  # Выполняются шаги #3 и #4
```

**available_steps** - доступные для выполнения:
```python
[2, 3, 4]  # Шаги #3, #4, #5 можно выполнить
```

### Логика в tool_selector:

```python
# Определяем доступные шаги (еще не выполненные)
available_steps = [i for i in range(len(plan)) if i not in completed_steps]

# Передаем в промпт
render_ctx = {
    "user_query": state["user_query"],
    "plan": plan,
    "current_step_index": current_step_index,
    "completed_steps": completed_steps,
    "available_steps": available_steps,
    ...
}

# Сохраняем executing_steps из ответа LLM
executing_steps = result.plan_steps
state["executing_steps"] = executing_steps
```

### Логика в refiner:

```python
# Если текущие шаги выполнены, добавляем их в completed_steps
if result.current_step_completed:
    for step_idx in executing_steps:
        if step_idx not in completed_steps:
            completed_steps.append(step_idx)
    completed_steps.sort()

# Находим следующий невыполненный шаг
next_step_index = next((i for i in range(len(plan)) if i not in completed_steps), len(plan))
```
