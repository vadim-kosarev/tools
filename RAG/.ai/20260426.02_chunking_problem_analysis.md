# Проблема чанкинга: Информация разбита между чанками

## Дата: 2026-04-26

## 🔍 Описание проблемы

**Симптом:**
- Exact search находит 50 чанков с термином "АРМ эксплуатационного персонала СОИБ КЦОИ"
- LLM отвечает: "документация не указывает конкретное ПО"

**Корневая причина:**
Информация о конкретном ПО находится в **соседних чанках**, которые не содержат ключевую фразу.

### Пример:

**Чанк 1 (найден exact search):**
```
На АРМ эксплуатационного персонала СОИБ КЦОИ установлены следующие
программные средства Лаборатории Касперского:
```

**Чанки 2-5 (НЕ найдены exact search):**
```
1) Консоль администрирования Kaspersky Security Center 14.2.0.26967
92) Агент администрирования Kaspersky Security Center 14.2.0.26967
93) Kaspersky Endpoint Security 12.1.0.506 для Windows
```

**Проблема:** LLM видит только чанк 1 (заголовок), но не видит чанки 2-5 (сам список ПО).

---

## ✅ Решения

### Стратегия A: `get_neighbor_chunks()` + включение якоря

**Использование:**
```python
# 1. Находим якорный чанк exact search
docs = store.exact_search("АРМ эксплуатационного персонала СОИБ КЦОИ", limit=1)
anchor = docs[0]

# 2. Получаем соседние чанки
neighbors = store.get_neighbor_chunks(
    source=anchor.metadata['source'],
    line_start=anchor.metadata['line_start'],
    before=10,
    after=10
)

# 3. ВАЖНО: Добавляем сам якорь к соседним чанкам!
all_chunks = neighbors + [anchor]
all_chunks = sorted(all_chunks, key=lambda d: d.metadata['line_start'])

# 4. Собираем полный текст
full_text = "\n".join(doc.page_content for doc in all_chunks)
```

**Преимущества:**
- ✅ Получаем контекст вокруг найденного термина
- ✅ Контролируем объём контекста (before/after параметры)

**Недостатки:**
- ❌ Нужно помнить включать сам якорь
- ❌ Может не захватить весь список если он длинный (>10 чанков)

---

### Стратегия B: `get_section_content()` - читать весь раздел

**Использование:**
```python
from kb_tools import create_kb_tools

tools = create_kb_tools(vectorstore, knowledge_dir)
get_section_content = tools[9]  # 10-й инструмент

result = get_section_content.invoke({
    "source_file": "Общее описание системы Приложение Г (ИИ-26).md",
    "section": "Решения по комплексу технических средств СЗВВК"
})
```

**Преимущества:**
- ✅ Гарантированно получаем ВЕСЬ раздел целиком
- ✅ Не теряем информацию из-за границ чанков
- ✅ Простая логика

**Недостатки:**
- ❌ Может быть избыточным если раздел очень большой
- ❌ Требует знания точного названия раздела

---

### Стратегия C: Комбинированная (рекомендуется для агентов)

**Алгоритм:**

1. **Exact search** - находим чанки с ключевым термином
2. **Извлекаем метаданные** - смотрим source + section
3. **get_section_content()** - читаем ПОЛНЫЙ раздел для каждого уникального (source, section)

**Пример:**
```python
# 1. Находим все чанки с термином
docs = store.exact_search("АРМ эксплуатационного персонала СОИБ КЦОИ", limit=50)

# 2. Извлекаем уникальные разделы
unique_sections = set()
for doc in docs:
    unique_sections.add((doc.metadata['source'], doc.metadata['section']))

# 3. Читаем каждый раздел полностью
for source, section in unique_sections:
    full_section = get_section_content.invoke({
        "source_file": source,
        "section": section
    })
    # Передаем full_section в LLM
```

**Преимущества:**
- ✅ Нет потерянной информации между чанками
- ✅ Автоматическое обнаружение релевантных разделов
- ✅ Масштабируется на множество разделов

**Недостатки:**
- ❌ Может быть большой объём данных если много уникальных разделов

## 🔧 Рекомендации для агентов

### rag_agent.py (многошаговый пайплайн)

**Текущая реализация:**
- Шаг 2: multi_retrieve (semantic + exact + regex)
- Шаг 3: enrich_with_neighbor_chunks (±N чанков вокруг)

**Проблема:**
- `enrich_with_neighbor_chunks()` использует `get_neighbor_chunks()`, но **не включает якорь**

**Решение:**
```python
# В функции enrich_with_neighbor_chunks():
neighbors = store.get_neighbor_chunks(source, line_start, before, after)

# ДОБАВИТЬ: включаем якорный чанк
all_chunks = neighbors + [anchor_chunk]  # <- ЭТО ВАЖНО!
all_chunks = sorted(all_chunks, key=lambda d: d.metadata['line_start'])
```

### rag_lc_agent.py (ReAct агент)

**Текущая реализация:**
- Агент выбирает инструменты самостоятельно
- Есть `get_neighbor_chunks` и `get_section_content`

**Проблема:**
- Агент может использовать только `exact_search` без расширения контекста
- Промпт должен явно указывать на необходимость расширения контекста

**Решение:**
Обновить system prompt:
```
ПРАВИЛО: Если exact_search находит заголовок списка / таблицы / 
перечисления, ОБЯЗАТЕЛЬНО используй один из инструментов расширения 
контекста:
  1. get_section_content(source_file, section) - для ПОЛНОГО раздела
  2. get_neighbor_chunks(source, line_start, after=15) - для 15 следующих чанков

Признаки заголовка:
  - "установлены следующие"
  - "включает в себя:"
  - "состоит из:"
  - номера списка (1), 2), 3)...) в начале
```

---

### rag_lg_agent.py (LangGraph итеративный)

**Текущая реализация:**
- Итеративный граф с узлами planner → tool_selector → analyzer → refiner
- analyzer извлекает данные из last_result

**Проблема:**
- tool_selector может выбрать exact_search
- Но не понимает, что нужно расширить контекст для полноты

**Решение:**
Добавить в analyzer логику автоматического обнаружения "неполных" результатов:
```python
# В analyzer node:
if "следующие" in last_result or "включает:" in last_result:
    # Это похоже на заголовок списка
    gaps.append("Найден заголовок, но не найден полный список")
    # refiner увидит gap и запросит get_section_content
```

## 📝 Выводы

1. **Проблема подтверждена:** Информация о ПО для АРМ СОИБ КЦОИ **ЕСТЬ в документации**

2. **Корневая причина:** Exact search находит заголовок, но не находит сам список (он в соседних чанках)

3. **get_neighbor_chunks() НЕ включает якорь** - это важная особенность реализации

4. **Решение:** Использовать:
   - `get_section_content()` для полных разделов (предпочтительно)
   - `get_neighbor_chunks() + [anchor]` для точечного расширения

5. **Для агентов:** Обновить промпты и логику для автоматического обнаружения "неполных" результатов
