## 2026-04-28 - Исправление пустых файлов xxx_llm_db в логах

### Проблема
В папке `logs/` создавались пустые файлы с именами типа `007_llm_db`, `008_llm_db` без расширения `.log`.

### Что исправлено
✅ **Замена недопустимых символов** в именах файлов:
- Символ `:` (двоеточие) заменяется на `_` (подчеркивание)
- Windows не позволяет использовать `:` в именах файлов

✅ **DB запросы теперь логируются как tool calls**:
- Было: `007_llm_db:semantic_search_request.log` → создавал пустой `007_llm_db`
- Стало: `007_tool_semantic_search_request.log` → создается корректный лог

✅ **Создан тест** `test_db_logging.py` для проверки

### Файлы
- `llm_call_logger.py` - исправлены методы `_write()` и `_write_streaming_header()`
- `test_db_logging.py` - тест
- `016_db_logging_fix.md` - подробное описание

### Очистка
```powershell
Remove-Item -Path "logs/*_llm_db" -Force
```

## 2026-04-27 - Исправление истории сообщений в LLM запросах

### Проблема
Секция `[MESSAGES]` в логах LLM запросов не содержала историю сообщений - только текущий запрос.

### Что исправлено
✅ **Все узлы теперь передают полную историю** в LLM вызовы:
- `action_node` - передает историю предыдущих шагов
- `observation_node` - передает историю + результаты инструментов  
- `refine_node` - передает всю цепочку рассуждений
- `final_node` - передает полный контекст для финального ответа

✅ **User сообщения сохраняются** в `state["messages"]`

✅ **Tool результаты** уже сохранялись и продолжают сохраняться

### Структура истории
```
plan_node:
  system → user → assistant

action_node: 
  [история] → user → assistant → tool_1 → tool_2 → ...

observation_node:
  [история] → user → assistant

refine_node:
  [история] → user → assistant

final_node:
  [история] → user → assistant
```
