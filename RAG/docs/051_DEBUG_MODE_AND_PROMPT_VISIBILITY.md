# Режим отладки промптов + Улучшение видимости вопроса пользователя

## Обзор изменений

В этой сессии были реализованы два важных улучшения:

1. **Режим отладки с ограничением по шагам** (`--steps N`)
2. **Улучшение видимости вопроса пользователя в промптах**

---

## Часть 1: Режим отладки (--steps)

### Проблема
Для отладки промптов нужно было видеть промежуточные шаги работы агента, не дожидаясь полного выполнения.

### Решение

Добавлен аргумент командной строки `--steps N`, который ограничивает выполнение N шагами (нодами графа).

#### Использование
```bash
# Выполнить только первые 2 шага
python rag_lg_agent.py --steps 2 "ваш вопрос"

# С подробным выводом
python rag_lg_agent.py --steps 3 --verbose "ваш вопрос"
```

#### Последовательность нод графа
1. **plan** - Построение плана действий
2. **action** - Выполнение инструментов
3. **observation** - Обработка результатов
4. **refine** - Принятие решения (продолжить или финализировать)
5. **action** (повтор) - Второй цикл, если нужно
6. **observation** (повтор)
7. **refine** (повтор)
8. **final** - Формирование финального ответа

#### Примеры

**Отладка только планирования (1 шаг):**
```bash
python rag_lg_agent.py --steps 1 "что такое СОИБ"
```
Выполнится только нода `plan`, можно посмотреть:
- Логи: `logs/001_llm_plan_request.log`, `logs/002_llm_plan_response.log`
- Сформированный план действий

**Отладка первого вызова инструментов (2 шага):**
```bash
python rag_lg_agent.py --steps 2 "что такое СОИБ"
```
Выполнится `plan` → `action`, можно посмотреть:
- Какие инструменты вызываются
- Логи запросов к LLM

#### Вывод в режиме отладки

При остановке до финальной ноды отображается:
```
================================================================================
Вопрос: что такое СОИБ
🔍 РЕЖИМ ОТЛАДКИ: лимит 2 шагов (нод графа)
Шагов: 1
Итераций: 1/2
Messages: 2
Tools executed: 0
================================================================================

================================================================================
🔍 ОТЛАДКА - СОСТОЯНИЕ НА МОМЕНТ ОСТАНОВКИ:
================================================================================

⚠️  Выполнение остановлено до формирования финального ответа

📊 Текущее состояние:
  - Plan: 3 шагов
  - Tool calls: 0
  - Observation: 0 символов
================================================================================
```

#### Реализация

**Изменения в коде:**
- `parse_args()` - добавлен аргумент `--steps`
- `run_query()` - использует `graph.stream()` вместо `graph.invoke()` в режиме отладки
- `print_result()` - специальный вывод для незавершенного состояния
- `main()` и `run_interactive()` - передают параметр `max_steps`

**Логирование:**
- В начале: "🔍 РЕЖИМ ОТЛАДКИ: выполнить максимум N шагов"
- Каждый шаг: "Шаг 1/2: нода 'plan'"
- Остановка: "🛑 ОСТАНОВКА: достигнут лимит N шагов"

### Польза для отладки

1. **Проверка промптов** - Можно посмотреть какие промпты отправляются в LLM
2. **Анализ ответов** - Как LLM интерпретирует инструкции
3. **Пошаговое выполнение** - Найти на каком шаге что-то идет не так
4. **Экономия времени** - Не ждать полного выполнения при отладке начальных шагов
5. **Тестирование изменений** - После правки промптов проверить первые шаги

## Часть 2: Улучшение видимости вопроса пользователя

### Проблема

LLM отвечала "пользователь не задал вопрос", хотя вопрос присутствовал в messages.

**Причина:**
- Системный промпт очень большой (600+ строк)
- Вопрос пользователя был в самом конце
- Вопрос не был визуально выделен
- Формат был простой: "Вопрос пользователя: {{ user_query }}"

### Решение

#### 1. Визуальное выделение вопроса

**ДО:**
```
Вопрос пользователя:
{{ user_query }}
```

**ПОСЛЕ (plan/user.md - главная страница):**
```
================================================================================
🎯 ВОПРОС ПОЛЬЗОВАТЕЛЯ (ГЛАВНАЯ ЗАДАЧА):
================================================================================

{{ user_query }}

================================================================================
```

**ПОСЛЕ (остальные ноды):**
```
🎯 **ВОПРОС ПОЛЬЗОВАТЕЛЯ:**
{{ user_query }}
```

#### 2. Напоминание в system промпте

Добавлено в `system_prompt_base.md`:
```markdown
**🎯 ГЛАВНОЕ:** Твоя задача — ответить на вопрос пользователя.
Вопрос будет в USER сообщении (отмечен эмодзи 🎯).
Внимательно прочитай его и работай над ЕГО решением.
```

#### 3. Напоминание в конце инструкций для plan

Добавлено в `plan/system.md`:
```markdown
⚠️ ВАЖНО: Вопрос пользователя будет в следующем сообщении (USER message). 
Внимательно прочитай его и сформируй план на его основе.
```

### Измененные файлы

1. ✅ `prompts/system_prompt_base.md` - добавлено напоминание в раздел "Контекст"
2. ✅ `prompts/plan/user.md` - яркое выделение вопроса с рамками
3. ✅ `prompts/plan/system.md` - напоминание в конце
4. ✅ `prompts/action/user.md` - эмодзи и жирный шрифт
5. ✅ `prompts/observation/user.md` - эмодзи и жирный шрифт
6. ✅ `prompts/refine/user.md` - эмодзи и жирный шрифт
7. ✅ `prompts/final/user.md` - эмодзи и жирный шрифт

### Сравнение ДО и ПОСЛЕ

#### ДО: Вопрос терялся
```
... (600 строк системного промпта) ...

[#1 USER]
Вопрос пользователя:
что такое smart monitor
```
❌ **Проблема:** Вопрос в самом конце, не выделен, легко пропустить

#### ПОСЛЕ: Вопрос заметен

**Plan нода (первый контакт):**
```
... (600 строк системного промпта) ...

⚠️ ВАЖНО: Вопрос пользователя будет в следующем сообщении (USER message). 
Внимательно прочитай его и сформируй план на его основе.

[#1 USER]
================================================================================
🎯 ВОПРОС ПОЛЬЗОВАТЕЛЯ (ГЛАВНАЯ ЗАДАЧА):
================================================================================

что такое smart monitor

================================================================================
```

✅ **Решено:** 
- Рамки 80 символов привлекают внимание
- Эмодзи 🎯 - визуальный маркер "цель"
- Текст "ГЛАВНАЯ ЗАДАЧА" подчеркивает важность
- Предупреждение в конце system промпта

**Action/Observation/Refine/Final ноды:**
```
[#1 USER]
🎯 **ВОПРОС ПОЛЬЗОВАТЕЛЯ:**
что такое smart monitor

... (остальной контекст) ...
```

✅ **Решено:** 
- Эмодзи 🎯 для быстрого поиска
- Bold текст выделяется
- Вопрос в начале user message

### Почему разный уровень выделения?

- **plan** - максимальное выделение (рамки 80 символов)
  - Причина: это первый контакт с вопросом, нужно максимально привлечь внимание
- **Остальные ноды** - среднее выделение (эмодзи + bold)
  - Причина: контекст уже известен, достаточно напоминания

### Измерение эффективности

**ДО изменений:**
```json
{
  "thought": "Пользователь не задал вопрос...",
  "plan": []
}
```

**ПОСЛЕ изменений:**
```json
{
  "thought": "нужно найти информацию о smart monitor",
  "plan": [
    "семантический поиск по термину smart monitor",
    "точный поиск если определение не найдено",
    "структурировать информацию"
  ]
}
```

## Итоги

### Режим отладки
- ✅ Можно выполнять только первые N шагов графа
- ✅ Логи сохраняются на каждом шаге
- ✅ Специальный вывод для незавершенного состояния
- ✅ Удобно для отладки промптов и логики агента

### Видимость вопроса
- ✅ Вопрос теперь визуально выделен во всех промптах

- ✅ Разный уровень выделения для разных этапов

### Совместное использование
Режим отладки + улучшенные промпты = мощный инструмент для:
- Быстрой проверки изменений в промптах
- Пошагового анализа работы агента
- Выявления проблем в понимании инструкций LLM
