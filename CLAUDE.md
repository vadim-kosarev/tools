## Генерация кода (если не указано иное, то...)
* всегда проверять актуальность кода в открытых источниках используемых библиотек и фреймворков
* использовать только актуальные версии библиотек и фреймворков
* избегать использования устаревших методов и функций
* при генерации кода стараться использовать современные стандарты и лучшие практики программирования
* методы и функции должны быть максимально типизированны (использовать type hints и Pydantic в Python)
* по возможности добавлять к методам тип возвращаемого значения
* во время написания кода придерживаться принципов чистого кода (Clean Code) и SOLID
* стараться писать код, который легко тестировать и поддерживать
* избегать избыточной сложности и дублирования кода (DRY)
* использовать асинхронное программирование там, где это уместно
* если не указано иное, использовать асинхронные методы и функции (async/await) для операций ввода-вывода и сетевых запросов
* не нужно проверять досконально решение после его реализации, если не указано иное
* не нужно проверять, применились ли изменения
* если есть изменения в коде, после внесения изменений обновлять документацию метода/класса/модуля (docstrings)

### Python-специфичные правила

#### Логирование
- ❌ **НЕ использовать** `print()` для вывода информации
- ✅ **Использовать** `logger.info()` и `logger.debug()`
- ✅ **Перед вызовом ключевого метода:** `logger.debug()` с описанием действия
- ✅ **После вызова ключевого метода:** `logger.info()` с результатом
- ✅ **JSON-сериализация:** если данные для логирования можно сериализовать в JSON - сериализовать в JSON

**Для красивого логирования с переносом строк:**
- ❌ **НЕ использовать** несколько вызовов `logger.info()` подряд
- ✅ **Использовать** символ `"\n"` для переносов строк внутри одного вызова `logger.info()`
- ✅ **Минимизировать** количество вызовов `logger.info()` до одного для логического блока
- ℹ️ **Таймстамп** будет добавлен автоматически в начале первой строки блока

**Пример (НЕПРАВИЛЬНО):**
```python
logger.info("=" * 80)
logger.info("Processing started")
logger.info("Item count: 10")
logger.info("Status: running")
logger.info("=" * 80)
```

**Пример (ПРАВИЛЬНО):**
```python
logger.info(
    f"{'=' * 80}\n"
    f"Processing started\n"
    f"Item count: 10\n"
    f"Status: running\n"
    f"{'=' * 80}"
)
```

**Пример:**
```python
logger.debug(f"Calling API with params: {json.dumps(params)}")
result = api_call(params)
logger.info(f"API call result: {json.dumps(result)}")
```

#### Структуры данных
- ❌ **НЕ использовать** `dict` для сложных структур данных
- ✅ **Использовать** Pydantic `BaseModel` для всех структур данных
- ✅ **Все запросы со сложной структурой** должны быть типизированы через Pydantic
- ✅ **Все данные** должны быть именованного типа, наследуемого от `pydantic.BaseModel`

**Пример:**
```python
from pydantic import BaseModel

class UserRequest(BaseModel):
    name: str
    age: int
    email: str

class ApiResponse(BaseModel):
    status: str
    data: dict
    timestamp: datetime
```

## 🛠️ Среда разработки

### Основные параметры
- **IDE:** JetBrains IDEA
- **Shell:** PowerShell (Windows) - команды для run_in_terminal должны быть PowerShell-совместимыми. ОСЬ - Windows.
- для новых терминалов, используемых агентом нужно вызывать команду
   ```chcp 65001 ; $env:PYTHONIOENCODING = 'utf-8'``` - только один раз на терминал, не каждый раз перед командой
- после этого один раз активировать виртуальное окружение из каталога проекта скриптом по полному пути:
   ```IDE_PROJECT_ROOT\.venv\Scripts\Activate.ps1 2>&1```

### PowerShell команды (важно!)
При использовании `run_in_terminal` tool:
- ✅ Разделитель команд: `;` (НЕ `&&`)
- ✅ Чтение файлов: `type` (НЕ `cat`)
- ✅ Использовать только PowerShell-совместимые команды
- ✅ Команды 'head' и 'tail' НЕ использовать, вместо них использовать:
  - `Get-Content -TotalCount n` для чтения первых n строк файла
  - `Get-Content -Tail n` для чтения последних n строк
- ✅ **ОБЯЗАТЕЛЬНО** добавлять `[Console]::Out.Flush()` в конце команды для гарантированного вывода результата

**Пример:**
```powershell
cd path; python script.py    # ✅ Правильно
cd path && python script.py  # ❌ Ошибка в PowerShell

# Для гарантированного вывода:
git commit -m "message" ; git log --oneline -1 ; [Console]::Out.Flush()  # ✅ Правильно
```

## 📁 Работа с файлами

### Создание файлов
- ❌ **НЕ создавать** тестовые скрипты/данные без явного запроса
- ❌ **НЕ создавать** `*.md` файлы без явного запроса
- ❌ **НЕ дублировать** существующие файлы проекта
- ✅ При создании файлов учитывать структуру проекта в IDEA

### Тестовые скрипты
- ❌ **НЕ генерировать** `test_*.py` без явного запроса
- ❌ **НЕ генерировать** `usage_example_*.py` без явного запроса
- ✅ Генерировать тестовые скрипты ТОЛЬКО если пользователь явно запросил

### Документация и отчёты
- если генеришь `SUMMARY_*.md`, `COMPLETE_*.md` и прочие файлы отчетов -- класть их в папку doc в рабочем каталоге
- если генеришь тесты, класть их в папку test в рабочем каталоге, вызывать из рабочего каталога
- ✅ **ОБЯЗАТЕЛЬНО обновлять** `README.md` в рабочем каталоге по мере выполнения работы
  - Добавлять описание новых функций/модулей
  - Обновлять инструкции по использованию
  - Документировать важные изменения
  - Актуализировать примеры кода
- ✅ Генерировать документацию ТОЛЬКО если пользователь явно попросил

#### Если отчет запрошен явно:
1. **Папка:** `.ai/`
2. **Формат имени:** `YYYYMMDD.NN_<название>.md`
   - `YYYYMMDD` - текущая дата
   - `NNN` - порядковый номер (трехзначный, с ведущими нулями)
3. **Порядковый номер:**
   - Читается из файла `.ai/version.txt`
   - Инкрементируется при создании нового отчета
   - Сохраняется обратно в `.ai/version.txt`

**Пример:**
```
.ai/20260215.01_api_integration_report.md
.ai/20260215.02_database_migration_report.md
.ai/version.txt  # содержит: 02
```
