# Immich: план отказоустойчивости (brightsky / starlight / luigi)

**Дата:** 2026-07-08
**Статус:** план, реализация не начата

## Повод

У brightsky (основной хост с Immich) сдох блок питания — полный простой до физической замены.
Данные (фото + Postgres) физически лежали на локальном диске brightsky (`K:/immich/upload`,
`K:/immich/postgres`) — единая точка отказа и по железу, и по хранилищу.

## Хосты и роли

| Хост | Роль | Железо |
|---|---|---|
| **brightsky** | Основной compute-хост Immich | Docker, GPU |
| **starlight** | Резервный compute-хост (иногда перезагружается для работы) | Docker, диски, нормальный CPU, GPU |
| **luigi** | Файлохранилище / NAS | Celeron 1.5GHz, несколько винтов — **только storage**, в compute не участвует |

Ограничения, определяющие архитектуру:
- ML / transcoding в Immich — GPU-bound, поэтому compute-нода обязана быть brightsky или starlight. luigi для контейнеров Immich не подходит.
- Postgres не переживает сетевую ФС под данными — нужна репликация, а не "тот же файл БД по SMB/NFS".

## Целевая архитектура — warm standby (active/passive), без полного кластера

Осознанно не полный HA (Swarm/Patroni/Keepalived+VIP): избыточная сложность для домашнего
single-user сетапа, плюс GPU-pinning всё равно не даёт контейнерам свободно шедулиться между
хостами. Цель — не бесшовно, а **разумный downtime** (минуты) при полном контроле над моментом
переключения.

### 1. Storage → непрерывная репликация файлов (реализовано частично)

Вместо переноса на общую сетевую шару luigi (это отдельная задача на будущее, пока не делаем) —
выбрана **непрерывная репликация файлов Resilio Sync** между локальными дисками хостов:

- brightsky: `K:\immich\upload\photos` (primary, read/write)
- starlight: `H:\immich\upload\photos` (реплика)

Уже настроено и работает. Важно проверить/зафиксировать:
- starlight подключён через **Read-Only ключ** Resilio (не read/write) — иначе случайные
  локальные изменения на starlight (например, ошибочные аплоады при неполном failover) улетят
  обратно на brightsky и испортят primary
- разные пути на хостах → нужен отдельный `.env`/`.env.starlight` с
  `UPLOAD_LOCATION=H:/immich/upload` (на brightsky — `K:/immich/upload`), по аналогии с уже
  существующим разделением `.env`/`.env.brightsky`
- **Уточнение (проверено по `docker-compose.prod.yml`):** том `database` смонтирован как
  `${UPLOAD_LOCATION}/postgres:/var/lib/postgresql/data` — то есть реальный путь к данным БД на
  brightsky это `K:\immich\upload\postgres` (вложен в `upload/`, рядом с `photos/`).
  `DB_DATA_LOCATION` в `.env` — неиспользуемая (мёртвая) переменная, реальный путь строится от
  `UPLOAD_LOCATION`. На starlight, соответственно: `H:\immich\upload\postgres` — один и тот же
  `UPLOAD_LOCATION=H:/immich/upload` покрывает и фото, и БД.
  **Папка `postgres/` не должна попадать под Resilio Sync** (у него в скоупе только `photos/`,
  так и оставить) — она заполняется исключительно через `pg_basebackup -R` и дальше управляется
  самим Postgres (WAL-репликация). Файловый синк поверх неё гарантированно побьёт реплику.

Синк не мгновенный (RPO — как и у Resilio/Syncthing вообще, минуты в худшем случае) — тот же
принцип "разумного downtime", что и у репликации БД.

Перенос на общую шару luigi остаётся как возможный будущий шаг (тогда обоим хостам не нужно
будет держать по полной локальной копии), но не блокирует текущий план.

### 2. Postgres → потоковая репликация brightsky → starlight

Не файловый дамп раз в сутки, а `pg_basebackup` / WAL streaming — standby на starlight всегда
почти актуален (RPO — секунды). При failover standby промоутится в primary локально на
starlight (без сети, без гонки за общим диском).

### 3. Failover — скриптованный, не автоматический

Автопромоушен реплики оставляем ручным/полуручным шагом, чтобы не словить split-brain, если
brightsky вдруг оживёт сам, пока starlight уже подхватил роль primary.

- health-check `/api/server/ping` на brightsky (можно повесить через существующий `cron-win`)
- при недоступности N минут — скрипт на starlight:
  1. promote Postgres standby → primary
  2. `docker compose up` на starlight с storage от luigi + локальным (уже promoted) Postgres

### 4. frpc — split per-service + естественный failover на уровне туннеля

Сейчас **один** `frpc`-клиент (`immich_frpc`, `immich/docker/docker-compose.prod.yml`,
физически на brightsky) тащит вообще всё: immich, `frigate-ui`, `luigi-grafana`,
`luigi-torrent`, `luigi-subsonic`, `starlight-vnc/flowise/ollama`, камеру. Пока brightsky был
мёртв — наружу отвалилось всё, включая то, что само по себе работало (luigi, starlight).

План:
- **luigi** — свой `frpc.ini` рядом со своими сервисами (grafana/torrent/subsonic/sync)
- **starlight** — свой `frpc.ini` (vnc/flowise/ollama)
- **immich-server** — прописан **в обоих** конфигах (brightsky и starlight) с одинаковым
  `remote_port=2283`. frps не даёt зарегистрировать дубль имени прокси — держит регистрацию
  тот, кто жив; как только активный отваливается по TCP, frps освобождает прокси и retry со
  второго хоста тут же его подхватывает. Автоматический failover **на уровне туннеля**, без
  Keepalived/VIP.
- подкрутить интервал retry в `frpc`, чтобы окно недоступности укладывалось в "разумный
  downtime"
- frigate/камера остаются прописаны там же, где физически стоит frigate — не зависят от того,
  какой хост сейчас держит Immich

## Текущее состояние (на момент написания)

- brightsky — жив и здоров, сейчас основной (primary) хост Immich.
- starlight — резервный хост: файлы фото синкаются через Resilio Sync
  (`K:\immich\upload\photos` → `H:\immich\upload\photos`, read-only ключ), Postgres streaming
  replication поднята и подтверждена (`pg_stat_wal_receiver.status = streaming`, слот
  `starlight_standby`). Standby-контейнер `immich_postgres` поднимается отдельно от полного
  стека: `docker compose -f docker-compose.prod.yml --env-file .env.starlight up -d database`.
- Из `docker-compose.prod.yml` убрана директива `env_file: - .env` у `immich-server`,
  `immich-machine-learning` и `database` (мешала однозначно понимать, какие переменные откуда
  берутся при разных `--env-file` на разных хостах). Для `immich-server` DB/Redis-креды,
  которые раньше приходили неявно через `env_file`, перенесены в явный `environment:`
  (`DB_HOSTNAME`, `DB_USERNAME`, `DB_PASSWORD`, `DB_DATABASE_NAME`, `REDIS_HOSTNAME`).
- После того как этот план реализован и обкатан — деплой всего стека переезжает на GitLab CI
  (пайплайны деплоя из gitlab).
- **Пробный запуск полного инстанса на starlight выполнен и подтверждён** (открыт и работает на
  `http://localhost:2283`). По пути найдено и решено:
  - `IMMICH_VERSION=release` — плавающий тег; `docker compose up` не перекачивает образ, если
    тег уже закэширован локально. На starlight был старый локальный кэш `release` (v2.4.1),
    из-за чего `immich-server` падал на `corrupted migrations` (миграции жёстко привязаны к
    версии сервера, brightsky крутит v3.0.1). Фикс: явный `docker compose pull immich-server`
    перед `up`. **Риск на будущее:** при реальном failover нужно обязательно делать `pull` перед
    поднятием стека на резервном хосте — иначе есть шанс словить ту же рассинхронизацию версий.
    Возможно, стоит в будущем пиновать `IMMICH_VERSION` на конкретный тег вместо `release`.
  - Заведён `docker-compose.starlight.override.yml` — на случай если `luigi` (внешние библиотеки
    `vkshare`/`o-homevideo`/`o-photos`/`yd-photos`/`google-takeout`) недоступен, он убирает эти
    монтирования только для starlight, не трогая `docker-compose.prod.yml`. В этот раз `luigi`
    удалось вернуть в сеть, override не потребовался, но файл оставлен как готовый fallback.
  - **Важно:** для этого теста standby был **промоутнут** (`pg_promote()`) — starlight стал
    отдельным live-primary, разошедшимся с brightsky. Чтобы вернуть настоящий warm standby,
    нужно пересидить (`pg_basebackup -R`) заново с нуля.

## Инструкция: сидинг standby и пробный запуск на starlight

Проверено вручную end-to-end 2026-07-09. Порядок действий (все команды — на starlight, если не
указано иное).

### 1. Сидинг standby (на чистом/пересеянном `H:\immich\upload\postgres`)

На **brightsky** (primary) — один раз, роль/слот уже созданы, повторно не нужно:
```powershell
docker exec immich_postgres psql -U postgres -d immich -c "CREATE ROLE replicator WITH REPLICATION LOGIN PASSWORD 'пароль';"
docker exec immich_postgres psql -U postgres -d immich -c "SELECT pg_create_physical_replication_slot('starlight_standby');"
docker exec immich_postgres psql -U postgres -d immich -c "ALTER SYSTEM SET max_slot_wal_keep_size = '10GB';"
docker exec immich_postgres psql -U postgres -d immich -c "SELECT pg_reload_conf();"
```
В `K:\immich\upload\postgres\pg_hba.conf` должна быть строка (адрес `all` — из-за NAT
Docker Desktop/WSL2 на starlight конкретный IP не работает, см. план выше):
```
host replication replicator all scram-sha-256
```
+ `SELECT pg_reload_conf();` после правки.

На **starlight** — `H:\immich\upload\postgres` должна быть пустой/не существовать:
```powershell
docker run --rm -e PGPASSWORD='пароль' -v H:/immich/upload/postgres:/var/lib/postgresql/data `
  ghcr.io/immich-app/postgres:14-vectorchord0.3.0-pgvectors0.2.0 `
  pg_basebackup -h 192.168.1.43 -p 5432 -U replicator -D /var/lib/postgresql/data -Fp -Xs -P -R -S starlight_standby
```
Проверка, что реально стримит:
```powershell
docker exec immich_postgres psql -U postgres -d immich -c "SELECT * FROM pg_stat_wal_receiver;"
```
`status` должен быть `streaming`.

### 2. Пробный/аварийный полный запуск на starlight

**Внимание:** этот шаг промоутит standby в live-primary — реальная репликация с brightsky после
этого прекращается (см. п.3 про пересидинг). Не делать на живом standby без необходимости.

```powershell
cd C:\dev\github.com\vadim-kosarev\tools\immich\docker

# 1. Промоушен (standby -> read-write)
docker exec immich_postgres psql -U postgres -d immich -c "SELECT pg_promote();"

# 2. Убедиться, что образ immich-server актуальный (IMMICH_VERSION=release — плавающий тег,
#    docker compose up НЕ перекачивает уже закэшированный локально образ!)
docker compose -f docker-compose.prod.yml --env-file .env.starlight pull immich-server

# 3. Поднять redis + immich-server (ML и frpc не нужны для базовой проверки)
docker compose -f docker-compose.prod.yml --env-file .env.starlight up -d redis immich-server
```
Если `luigi` (внешние библиотеки vkshare/o-homevideo/o-photos/yd-photos/google-takeout)
недоступен — контейнер не поднимется (ошибка монтирования CIFS-тома). Обходной путь —
`docker-compose.starlight.override.yml`:
```powershell
docker compose -f docker-compose.prod.yml -f docker-compose.starlight.override.yml --env-file .env.starlight up -d immich-server
```

Открыть `http://localhost:2283` — должен показать логин и библиотеку.

### 3. Погасить тестовый инстанс и вернуть настоящий standby

```powershell
docker compose -f docker-compose.prod.yml --env-file .env.starlight stop immich-server redis
```
`immich_postgres` при этом остаётся промоутнутым (не standby). Чтобы вернуть настоящий warm
standby — снести данные и пересидеть заново (шаг 1):
```powershell
docker compose -f docker-compose.prod.yml --env-file .env.starlight stop database
docker compose -f docker-compose.prod.yml --env-file .env.starlight rm -f database
Remove-Item -Recurse -Force H:\immich\upload\postgres
# дальше — пересидинг из п.1
```
