# Immich v2.7.5 -> v3.0.1: миграция pgvecto-rs -> vchord

**Дата:** 2026-07-04  
**Версия до:** v2.7.5  
**Версия после:** v3.0.1

## Проблема

После `docker compose pull` + `docker compose up -d` сервер `immich_server` уходил в бесконечный restart:

```
Error: No vector extension found. Available extensions: vchord, vector
```

Immich v3.0.0 отказался от поддержки `pgvecto-rs` (`vectors` extension).  
Теперь поддерживаются только `pgvector` (`vector`) и `vchord`.  
Postgres-образ в compose был `tensorchord/pgvecto-rs:pg14-v0.2.0` — старый.

## Диагностика

В БД `immich` было установлено `vectors 0.2.0` (pgvecto-rs) — его новый сервер не видит.

## Решение

### 1. Изменить образ postgres в `docker-compose.prod.yml`

```yaml
# было:
image: tensorchord/pgvecto-rs:pg14-v0.2.0@sha256:739cdd...

# стало:
image: ghcr.io/immich-app/postgres:14-vectorchord0.3.0-pgvectors0.2.0
```

Переходный образ содержит оба расширения: старый `vectors` (для чтения данных) и новый `vchord`.

### 2. Добавить `vchord.so` в `shared_preload_libraries`

```yaml
# было:
postgres -c shared_preload_libraries=vectors.so ...

# стало:
postgres -c shared_preload_libraries=vchord.so,vectors.so ...
```

### 3. Пересоздать контейнер БД и установить расширение

```powershell
docker compose -f docker-compose.prod.yml up -d database
docker exec immich_postgres psql -U postgres -d immich -c "CREATE EXTENSION IF NOT EXISTS vchord CASCADE;"
```

### 4. Перезапустить сервер

```powershell
docker compose -f docker-compose.prod.yml up -d immich-server
```

Сервер автоматически перестраивает индексы при старте:

```
Reindexing clip_index (This may take a while, do not restart)  — 87753 rows
Reindexing face_index (This may take a while, do not restart)  — 83407 rows
```

Заняло ~8 минут. После завершения сервер перешёл в статус `healthy`.

## Итог

Все контейнеры работают, данные сохранены. Бэкап перед миграцией был актуальным (04.07.2026 05:02, 1.1 GB).
