# Face Indexer — задача

## Цель

Скрипт `face-indexer.py`, запускаемый периодически по cron, который:

1. Ходит в Immich через **REST API**.
2. Ищет людей (persons) из заданного списка.
3. Для каждого человека создаёт или обновляет **альбом**, названный по имени этого человека, и приводит его состав в точное соответствие результату поиска по этой персоне (полная синхронизация, не только добавление).

**Зачем:** альбом (в отличие от самой персоны в Immich) можно расшарить между пользователями Immich или сделать публичным по ссылке. Автоматическая синхронизация "персона -> альбом" даёт возможность делиться фотографиями конкретного человека с другими людьми, не открывая доступ ко всей библиотеке.

## Конфигурация

Рядом со скриптом, по аналогии с `face-search` / `face-finder`:

- **`.env`** — координаты Immich:
  - `IMMICH_URL` — базовый адрес Immich API
  - `IMMICH_API_KEY` — API-ключ (User Settings -> API Keys)
- **`.env.json`** — список людей, за которыми нужно следить. Каждый элемент списка может быть задан в одном из двух форматов (на выбор, свободно):
  - **ссылка** на персону из Immich UI, например: `http://brightsky:2283/people/440d8181-de84-43e3-8e66-3125b2889a2c?previousRoute=%2Fpeople` — person id извлекается из пути `/people/{id}`, query-параметры игнорируются;
  - **имя** персоны как строка, например: `"Vsevolod Fadeev"` — разрешается через `GET /api/people` по совпадению имени.

## Запуск

Docker-образ (`Dockerfile`), запущенный как сервис `face-indexer` в `immich/docker/docker-compose.prod.yml`, с cron **внутри контейнера** (пакет `cron`, файл `crontab` → `/etc/cron.d/face-indexer`, `entrypoint.sh` пробрасывает нужные env-переменные в `/etc/environment` для cron-джобы и запускает `cron -f`).

- Расписание: раз в сутки, `30 3 * * *`.
- Конфиг (`.env.json`) и логи (`logs/`) смонтированы volume'ами с хоста — редактируются без пересборки образа.
- **Без `depends_on: immich-server`**: сервис ходит в Immich только по REST через `IMMICH_URL`/`IMMICH_API_KEY`, поэтому не обязан быть частью одного и того же compose-стека или хоста — может быть развёрнут где угодно, лишь бы был сетевой доступ до Immich.
- Реально Immich крутится на отдельной машине `brightsky` (не на этом dev-хосте). Контейнер `face-indexer` поднят локально (dev-хост, Docker Desktop) и ходит на `brightsky` по сети через override `FACE_INDEXER_IMMICH_URL=http://192.168.1.43:2283` в `immich/docker/.env` (по аналогии с уже существующим `FACE_FINDER_IMMICH_URL`). На самом `brightsky` этот override не нужен — там сработает дефолт `http://immich-server:2283` (docker network).

## Логика (черновик)

1. Прочитать `.env` (URL/API key) и `.env.json` (список персон для отслеживания).
2. `GET /api/people` — получить всех персон Immich, сопоставить с именами из конфига. Если имени соответствует несколько person id (дубли в Immich) — брать **все** совпавшие id (проще, чем выбирать "первого", и корректнее: под одним именем в альбоме должны оказаться все, кого Immich считает этим человеком).
3. Для каждой найденной персоны/группы id с одним именем:
   - получить текущий набор assets по поиску Immich для всех id разом (например `POST /api/search/metadata` с `personIds: [id1, id2, ...]`) — это "целевой" набор;
   - проверить, существует ли альбом с именем персоны (`GET /api/albums`);
   - если нет — создать (`POST /api/albums`);
   - получить текущий состав альбома и сравнить с целевым набором:
     - assets, которые есть в поиске, но нет в альбоме — добавить (`PUT /api/albums/{id}/assets`);
     - assets, которые есть в альбоме, но нет в поиске — удалить из альбома (`DELETE /api/albums/{id}/assets`).
   - альбом должен **точно соответствовать** результату поиска по персоне на момент запуска: появилось новое фото в поиске — появляется в альбоме; фото перестало находиться по персоне (переклассификация лица и т.п.) — удаляется из альбома. Удаление — только из альбома, не самого asset.
4. Логирование по образцу `face_search_web.py` (module `logging`, `LOG_LEVEL` из `.env`).

**Важно (проверено на Immich 3.0.1):** `GET /api/albums/{id}` в этой версии не отдаёт список assets, только `assetCount`. Текущий состав альбома получаем так же, через `POST /api/search/metadata` с `albumIds: [id]` (тот же endpoint, что и для поиска по персоне).

## Реализация

- `face-indexer.py` — готов. CLI: `sync [--dry-run]` и `list` (резолвит персон из `.env.json` без записи), без аргументов — печатает help и выходит с кодом 0.
- Клиент Immich (`ImmichClient`) ходит напрямую, игнорируя системный `HTTP_PROXY`/`HTTPS_PROXY` (`trust_env=False`) — Immich всегда локальный хост.
- Проверено вживую (`sync --dry-run`) против `brightsky:2283`: все 9 персон из `.env.json` резолвятся, состав existing-альбомов сравнивается корректно (найдены реальные расхождения — несколько десятков assets на удаление, накопившихся с момента последнего ручного запуска старого `immich/scripts/update_persons_albums.py`, который только добавлял и не удалял).
